Self Explanations for Final Report
==================================

Question
--------
Stage 1 flattens the latest 20 x 13 bucket window to 260 features and trains an XGBoost regressor. Can you explain this in detail? You wrote this in the report.

Answer
------
This means TurboTest Stage 1 does not feed the whole 10-second trace directly into XGBoost.

Our data is bucketed every 100 ms. So:

20 buckets x 100 ms = 2000 ms = 2 seconds

At each decision time, Stage 1 takes the latest 20 observed buckets, where each bucket has 13 network features:

latest window shape = [20, 13]
20 time buckets x 13 features per bucket

XGBoost does not accept a 2D time-series tensor directly, so we flatten this matrix into one vector:

[20, 13] -> [260]
because 20 x 13 = 260

Example conceptually:

bucket 1:  13 features
bucket 2:  13 features
...
bucket 20: 13 features

flattened vector:
[f1_bucket1, f2_bucket1, ..., f13_bucket1,
 f1_bucket2, ..., f13_bucket2,
 ...
 f1_bucket20, ..., f13_bucket20]

Then XGBoost learns:

input:  latest 2 seconds of normalized network measurements, as 260 numbers
output: predicted final throughput of the full speed test in Mbps

So at 2.5 seconds, 3.0 seconds, 3.5 seconds, etc., it keeps forming the latest 2-second window and predicts what the final full-test throughput would be if the test had continued to completion.


Question
--------
Stage 2 uses 20 decision points at 500 ms stride, computes a permanent-safe suffix oracle from Stage 1 errors, and trains an 8-layer Transformer stop policy. The frozen epsilon = 10% baseline uses threshold 0.25. Can you explain this in detail? You wrote this in the report.

Answer
------
Stage 2 is the early-stop decision model. It does not predict throughput directly. Stage 1 already gives a throughput prediction at each possible stopping time. Stage 2 decides whether that prediction is safe enough to stop the test.

The trace has 100 buckets, and each bucket is 100 ms:

100 buckets x 100 ms = 10 seconds

Stage 2 does not make a stop decision at every 100 ms bucket. It makes decisions every 500 ms. Since:

500 ms = 5 buckets

the decision end buckets are:

4, 9, 14, 19, ..., 99

That gives exactly 20 decision points:

10 seconds / 0.5 seconds = 20 decisions

At each decision point d, Stage 1 has already produced a predicted final throughput:

y_pred_mbps[d]

The dataset also knows the true final throughput from the completed test:

y_true_mbps

So for every decision point, we compute the relative error:

relative_error[d] = abs(y_pred_mbps[d] - y_true_mbps) / y_true_mbps

For epsilon = 10%, a decision is instantaneously safe if:

relative_error[d] <= 0.10

But TurboTest does not train Stage 2 using only instantaneous safety. A single decision can be accidentally safe even if later predictions become unsafe again. To avoid labeling unstable points as safe stops, we use a permanent-safe suffix oracle.

The permanent-safe suffix oracle finds the first decision d* such that every decision from d* until the end remains safe:

safe[d*] = 1
safe[d* + 1] = 1
...
safe[last] = 1

Then Stage 2 labels are:

stop_label[d] = 0 for all d < d*
stop_label[d] = 1 for all d >= d*

If no such suffix exists, then the test has no safe early stop and all stop labels are 0.

Example:

instantaneous safe sequence:

[0, 0, 1, 0, 1, 1, 1]

Decision 2 is safe, but it is not permanently safe because decision 3 becomes unsafe again. The permanent-safe suffix starts at decision 4:

stop labels:

[0, 0, 0, 0, 1, 1, 1]

The Stage 2 Transformer is trained to predict these stop labels. It receives the normalized trace prefix available at each decision time and outputs a stop probability:

input:  trace prefix up to current decision
output: stop probability

The baseline Stage 2 model uses:

8 Transformer layers
8 attention heads
d_model = 128
feed-forward dimension = 512
dropout = 0.1

The frozen epsilon = 10% baseline means we treat the epsilon=10 trained Stage 2 model as our valid reproduced TurboTest baseline. Other epsilon baselines are not used because their datasets were stale.

The threshold 0.25 is not the epsilon. Epsilon defines what counts as safe:

epsilon = 10% means prediction error must be within 10%.

The threshold controls when the trained classifier actually stops:

if stop_probability >= 0.25:
    stop the test
else:
    continue

So threshold 0.25 means the model is allowed to stop when its predicted stop probability is at least 25%. This threshold was selected on validation data to balance accuracy and time savings.


Question
--------
What is FMNet-v3?

Answer
------
FMNet-v3 is the name we gave to our third foundation-model architecture for speed-test traces. It is not the original TurboTest model. It is our foundation encoder used in the later experiments.

FMNet means Foundation Model Network. Version 3 was designed after earlier patch-based foundation models did not preserve enough fine-grained timing information.

The key idea of FMNet-v3 is:

input:  full normalized speed-test trace [100, 13]
output: hidden representation for every 100 ms bucket [100, d_model]

It uses bucket-level causal attention:

- bucket-level means it keeps all 100 original 100 ms buckets instead of compressing them into 5-bucket patches.
- causal means bucket t can only attend to buckets <= t, not future buckets.

This is important for early stopping because a prediction at 2.5 seconds must not use information from 5 seconds or 10 seconds.

The final two-stage run used FMNet-v3 with:

- input features: 13
- buckets: 100
- Transformer layers: 10
- attention heads: 8
- model width: 256
- feed-forward width: 1024
- dropout: 0.15

FMNet-v3 was pretrained with a causal next-bucket prediction task:

given buckets 0..t, predict bucket t+1

After pretraining, we reused the encoder for downstream tasks:

1. Stage 1 prefix throughput prediction
2. Stage 2 early-stop policy
3. final throughput prediction

In the final two-stage foundation model, FMNet-v3 first converts the raw trace into hidden states. Then we gather hidden states at decision buckets:

4, 9, 14, ..., 99

These decision hidden states are used by the Stage 1 throughput head and then by the Stage 2 policy module.

So in simple words: FMNet-v3 is our causal Transformer foundation encoder for speed-test time series.
