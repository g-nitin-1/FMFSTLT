CRITICAL BUG: XGBoost Version Mismatch
- Diagnosis confirmed by direct code inspection and prediction comparison.
- Stage 1 XGBoost model was trained on Windows PowerShell.
  - XGBoost version used: 3.2.0 (confirmed from model JSON header: `"version": [3, 2, 0]`)
- Stage 2 datasets for epsilon = 5, 15, 20, 25, 30, 35 were built on WSL Linux.
  - XGBoost version on WSL: 3.0.5
  - XGBoost 3.0.5 cannot correctly load models saved by XGBoost 3.2.0
  - result: Stage 1 predictions are completely wrong (mean ~ -250 Mbps instead of ~+20 Mbps)
- Stage 2 dataset for epsilon = 10 was built on Windows PowerShell.
  - XGBoost version 3.2.0 matched training: predictions are correct
  - epsilon = 10 dataset and transformer are unaffected and valid
- Cascade of damage for epsilon = 5, 15, 20, 25, 30, 35:
  - corrupt Stage 1 predictions -> relative errors of 100x to 17000x
  - instantaneous_safe = 0 for all decision points
  - permanent-safe suffix oracle never fires -> stop_label = 0 everywhere
  - transformer trained on all-zero labels -> learned "never stop"
  - threshold sweep shows near-zero within_epsilon_rate for all thresholds
- Verified by:
  - recomputing Stage 1 predictions on WSL for a known test: got -260 Mbps vs correct +10.6 Mbps
  - checking oracle_stop_over_tests across epsilon values:
    - epsilon = 10: 79.65% (correct, built on Windows)
    - epsilon = 15: 0.08% (wrong, built on WSL)
    - epsilon = 35: 4.48% (wrong, built on WSL)
    - expected: rate must increase monotonically with epsilon; the actual values violate this

Foundation Model v1 Architecture
--------------------------------
- Input:
  - normalized trace tensor: `[100, 13]`
  - observed bucket mask: `[100]`
- Patchification:
  - 5 buckets per patch
  - `[100, 13] -> [20, 5, 13] -> [20, 65]`
  - patch is valid if at least one original bucket in the patch was observed
  - invalid patches are masked in the Transformer
- Encoder:
  - CLS token
  - patch embedding MLP: `65 -> 128 -> 256`
  - 8-layer Transformer encoder
  - 8 attention heads
  - `d_model = 256`
  - `ff_dim = 1024`
  - dropout `0.1`
- Heads implemented:
  - masked-patch reconstruction head
  - throughput regression head
  - epsilon=10 early-stop classifier head
  - speed-tier classifier head

Why Foundation v1 Did Not Beat Stage 2
--------------------------------------
- Stage 2 baseline uses a very direct supervised signal:
  - Stage 1 prediction at every decision point
  - relative error at every decision point
  - permanent-safe suffix oracle labels
  - full-history prefix at the decision point
- Foundation v1 early-stop head does not explicitly model Stage 1-like prefix throughput prediction inside the stopping decision.
- Foundation v1 uses a single CLS embedding for each prefix and a BCE stop head.
- That makes it a useful representation learner, but the early-stop head is not structurally aligned with the policy:
  - it is not forced to predict throughput at every decision
  - it is not forced to estimate uncertainty/confidence
  - it is not trained with a direct cost for "stop wrong" vs "wait too long"
- Therefore:
  - throughput head can beat Stage 1 XGBoost on val/test
  - early-stop head can become competitive after threshold sweep
  - but it does not reliably beat Stage 2's specialized decision policy
- Separate robustness issue:
  - foundation v1 throughput robustness MAE/RMSE is worse than XGBoost:
    - XGBoost robustness MAE/RMSE: `35.23` / `173.52`
    - foundation robustness MAE/RMSE: `38.76` / `245.63`
  - this is the main weakness in the v1 result.
  - val/test gains are real, but robustness is the time-shifted Feb-Mar 2025 holdout.
  - adding more capacity alone can make this worse, so the next model must include explicit regularization or augmentation.
- Separate threshold issue:
  - v1 early-stop becomes competitive only after threshold sweep.
  - the policy-selected ablation gets high within-epsilon correctness by waiting too long.
  - v2 should aim for calibrated stop probabilities where threshold `0.50` is meaningful.
  - threshold sweeps will still be reported as secondary Pareto analysis, but the primary model should not depend on a fragile threshold search to look good.

Foundation v1.5 Gate Before Full v2
-----------------------------------
- Do not jump directly into a large redesign unless the cheap intermediate test fails.
- v1.5 hypothesis:
  - v1 lost Stage 2 mostly because the stop head was not jointly trained with per-decision throughput and uncertainty.
  - if that is true, adding per-decision auxiliary heads to the current patch encoder should close much of the early-stop gap.
- v1.5 architecture:
  - keep the existing patch encoder scale:
    - patch size: `5` buckets
    - 20 decision-aligned patch tokens
    - `d_model = 256`
    - 8 heads
    - 8 layers
  - remove dependence on only one CLS output for early stopping.
  - produce an output for every decision index `d in 0..19`.
  - decision `d` corresponds to buckets `[0, 5d+4]`.
  - train heads at every decision:
    - prefix throughput mean head
    - prefix throughput uncertainty head
    - stop logit head
    - optional speed-tier head from the final pooled representation
- v1.5 causality:

  - decision token `d` cannot attend to patch tokens `> d`.
  - this prevents future-patch leakage while allowing one forward pass for all 20 decisions.
- v1.5 evaluation purpose:
  - if v1.5 beats or matches Stage 2 epsilon=10, full v2 may be unnecessary.
  - if v1.5 still loses Stage 2, full v2 is justified because the failure is likely in the representation and temporal aggregation, not only the head design.

Foundation Model v2 Proposal
----------------------------
- Goal:
  - one foundation model that beats Stage 1 throughput prediction and Stage 2 early-stopping policy.
  - it must also address the v1 robustness regression.
- Core idea:
  - move from a single CLS-per-prefix classifier to an all-decision causal multi-task sequence model.
  - train throughput, uncertainty, and stop behavior at every decision point.
- Input:
  - normalized trace tensor: `[100, 13]`
  - observed bucket mask: `[100]`
  - no validation/test/robustness examples used during pretraining or training.
- Bucket-level stem:
  - chosen design: causal 1D convolution over bucket tokens.
  - channels:
    - input: `13`
    - hidden/output: `128`
  - kernel size: `5`
  - stride: `1`
  - padding: left-only causal padding, no right padding.
  - activation: GELU.
  - normalization: LayerNorm after channel projection.
  - dropout: `0.15`.
  - causality rule:
    - bucket representation at time `t` can use buckets `<= t` only.
    - it must not use buckets `> t`.
  - reason:
    - local temporal mixing is useful, but bidirectional bucket attention before decision aggregation would leak future information.
- Decision-token aggregation:
  - chosen design: end-bucket representation.
  - decision token `d` is the causal bucket representation at bucket `5d+4`.
  - this exactly matches Stage 2 decision timing.
  - no mean pooling across future buckets is allowed.
  - optional later ablation:
    - concatenate a causal mean over buckets `[5d, 5d+4]`.
    - this is not part of the first v2 implementation.
- Decision Transformer:
  - input tokens: 20 decision tokens.
  - causal mask:
    - decision `d` attends only to decisions `<= d`.
  - initial configuration:
    - `d_model = 256`
    - `num_heads = 8`
    - `num_layers = 6`
    - `ff_dim = 1024`
    - dropout `0.15`
  - reason for 6 layers:
    - v1 used 8 layers and already overfit robustness.
    - v2 starts slightly smaller with stronger regularization.
- Heads:
  - prefix throughput mean head:
    - predicts log throughput at every decision.
  - prefix throughput uncertainty head:
    - predicts log variance at every decision.
  - stop-policy head:
    - predicts stop probability at every decision.
  - full-trace throughput head:
    - uses decision token 19 to report Stage 1-comparable throughput metrics.
  - speed-tier head:
    - uses decision token 19 for downstream speed-grade classification.
- Uncertainty formulation:
  - chosen design: heteroscedastic Gaussian negative log likelihood on log throughput.
  - for each supervised decision:
    - model predicts `mu_d` and `log_var_d`
    - target is `log1p(y_true_mbps)`
    - loss is `0.5 * exp(-log_var_d) * (target - mu_d)^2 + 0.5 * log_var_d`
  - reason:
    - uncertainty becomes directly tied to throughput error, and the stop head can learn to avoid low-confidence early stops.
- Stop and policy losses:
  - base stop loss:
    - weighted BCE between predicted stop probability and frozen epsilon=10 `stop_label[d]`.
  - policy cost term:
    - define `safe_d := instantaneous_safe_window[d]` from the epsilon=10 Stage 2 dataset.
    - define `p_stop_d` as predicted stop probability.
    - define `elapsed_norm_d = (5d+5) / 100`, the fraction of the 10-second trace consumed.
    - `wrong_stop_cost = p_stop_d * (1 - safe_d)`
    - `wait_cost = (1 - p_stop_d) * safe_d * elapsed_norm_d`
    - `policy_loss = lambda_wrong * wrong_stop_cost + lambda_wait * wait_cost`
  - important semantic distinction:
    - `stop_label[d] = 1` means decision `d` is at or after the permanent-safe suffix oracle point.
    - `instantaneous_safe_window[d] = 1` means decision `d` is safe at that decision only.
    - therefore a decision can be instantaneously safe but still have `stop_label[d] = 0`.
    - this is intentional: BCE preserves the conservative suffix-oracle target, while policy loss teaches calibration around instantaneous safety and waiting cost.
  - initial weights:
    - `lambda_wrong = 2.0`
    - `lambda_wait = 0.2`
  - reason:
    - stopping when unsafe is much worse than waiting one more decision.
    - waiting too long is still penalized so the model does not become the conservative policy-selected v1 ablation.
- Multi-task loss weighting:
  - first implementation uses fixed weights, not GradNorm or learned task weights.
  - initial total loss:
    - prefix throughput NLL: `1.0`
    - full-trace throughput NLL: `0.5`
    - stop BCE: `1.0`
    - policy cost: `0.5`
    - speed-tier cross entropy: `0.2`
  - reason:
    - fixed weights make the first result interpretable.
    - adaptive loss balancing can be a later ablation if the fixed-weight model is unstable.
- Robustness regularization:
  - dropout increased from v1 `0.10` to `0.15`.
  - AdamW weight decay: `0.02`.
  - optimizer change is deliberate:
    - v1 used Adam with no weight decay.
    - v2 uses AdamW so weight decay is decoupled from gradient scaling.
  - valid patch/decision dropout during training:
    - randomly drop 10% of valid decision tokens from attention.
    - never drop all valid context for a sample.
  - bucket feature dropout during training:
    - randomly zero 5% of observed bucket vectors before the causal stem.
    - this happens after train-only normalization, so zero means normalized mean-substitution, not raw feature zero.
    - keep the observed bucket mask unchanged so the model learns missing/noisy measurements.
  - date/tier balance:
    - keep shard-level shuffling.
    - if robustness remains weak, add date-balanced sampling as the next ablation.
- Evaluation protocol:
  - primary early-stop report:
    - threshold fixed at `0.50`.
    - this tests whether the policy head is calibrated.
  - secondary early-stop report:
    - sweep thresholds from `0.01` to `0.99`.
    - report Pareto frontier against frozen Stage 2 epsilon=10.
  - Stage 1 comparison:
    - report val/test/robustness MAE and RMSE against frozen XGBoost.
    - throughput metrics are computed in Mbps after applying `expm1(mu)` to log-throughput predictions.
  - Stage 2 comparison:
    - report window F1, within-epsilon policy rate, emitted-stop rate, mean/median stop time, and mean/median savings.
- Success criteria:
  - Stage 1:
    - beat XGBoost val/test MAE.
    - robustness MAE must improve over v1, and ideally beat XGBoost.
  - Stage 2:
    - at threshold `0.50`, match Stage 2 within-epsilon rate within `0.005` while improving mean savings by at least `250 ms`, or
    - on the threshold sweep, strictly dominate Stage 2 at one operating point with same-or-better within-epsilon rate and same-or-better savings.
  - Robustness guardrail:
    - do not accept a model that improves val/test but worsens robustness relative to v1.

Project-Level Conclusion After Pure Foundation Experiments
----------------------------------------------------------
- Current evidence does not support the claim that a pure raw-trace foundation model can completely beat the TurboTest pipeline within this project timeline.
- What worked:
  - the v1 foundation throughput regressor improves Stage 1 final-throughput regression on val/test compared with XGBoost.
  - this is a valid foundation-model contribution because it uses self-supervised trace pretraining followed by supervised fine-tuning.
- What did not work:
  - pure raw-trace foundation models did not improve Stage 2 early-stop policy behavior.
  - v1/v1.5 stop heads either stopped too aggressively with poor correctness or avoided stopping.
  - v2 causal bucket-stem variants failed the prefix-throughput gate.
  - v2.3 could not even mimic the XGBoost prefix teacher.
- Interpretation:
  - TurboTest is difficult to beat end-to-end because it separates the problem into a very strong prefix regressor plus a task-specific stopping policy.
  - the Stage 2 Transformer benefits from XGBoost prefix predictions that are already highly accurate at decision time.
  - asking a pure foundation model to learn both prefix throughput estimation and stopping policy from raw buckets is harder than the current architecture can handle.
- Project framing:
  - the honest result is not "foundation models replace TurboTest completely."
  - the honest result is: foundation pretraining helps final-throughput regression, but full early termination still benefits from TurboTest's task-specific prefix-regression structure.
- Next project direction:
  - move from pure replacement to hybrid foundation-policy modeling.
  - use TurboTest/XGBoost prefix predictions and uncertainty/error features as explicit inputs.
  - optionally combine them with foundation trace embeddings.
  - learn a better stopping policy on top of the strong TurboTest prefix representation.
- This remains aligned with the project proposal:
  - it evaluates whether foundation pretraining provides measurable benefits.
  - it records where pure foundation learning helps and where it fails.
  - it motivates a more realistic foundation-assisted early-termination model instead of forcing an unsupported pure replacement claim.

Project Direction Update
------------------------

- Goal: a pure foundation model that beats both Stage 1 (throughput) and Stage 2 (early stop).
- If a budgeted set of v3 experiments (A, B, C) all fail, we revert to the hybrid plan.

v3 Architecture
---------------
- File: `scripts/foundation_model_v3.py`.
- Reasoning behind the design choices:
  - v1 patch encoder + CLS pool: 5-bucket compression destroyed information at decision time.
  - v2 causal patch stem: 5-bucket compression carried over; each decision token saw only 5 raw buckets through one conv layer.
  - v2.3 distillation failure (foundation could not even mimic XGBoost teacher): confirmed information bottleneck, not just a loss/training problem.
  - v3 removes the patch step entirely: 100 bucket tokens, no compression, causal attention.
- Architecture:
  - bucket projection: linear `13 -> 256`.
  - learned positional embedding `[1, 100, 256]`.
  - 8-layer Transformer encoder, 8 heads, ff=1024, dropout=0.15, norm_first=True.
  - explicit causal mask: bucket `t` can only attend to buckets `<= t` (verified by direct test in CPU smoke run).
  - per-decision heads applied at end-buckets `{4, 9, ..., 99}` (matches Stage 2 decision schedule):
    - throughput-mu MLP `256 -> 256 -> 1` (predicts log1p throughput).
    - throughput-logvar MLP `256 -> 256 -> 1` (heteroscedastic uncertainty).
    - stop-logit MLP `256 -> 256 -> 1` (BCE).
  - sequence-level heads at last bucket position:
    - final throughput head reuses the same `256 -> 256 -> 1` shape.
    - speed-tier head `256 -> 256 -> 5`.
  - pretraining head: next-bucket MLP `256 -> 256 -> 13`.
- Total params at default config: `6.68M`.

v3 Pretraining
--------------
- File: `scripts/pretrain_foundation_v3.py`.
- Self-supervised objective: causal next-bucket prediction.
- Strict scope: only the `720,000` Stage 1 train UUIDs (no val/test/robustness leakage).
- Loss: SmoothL1 between predicted bucket `t+1` and observed bucket `t+1`, masked by `bucket_mask`.
- Hyperparameters:
  - 5 epochs.
  - batch size 256.
  - AdamW, lr `3e-4`, weight decay `0.02`, warmup 500 steps.
  - feature dropout `5%` (zero a small fraction of bucket vectors before encoding).
  - gradient clipping `1.0`.
- Run on WSL CUDA:
  - command: `python3 /mnt/e/fmfstlt/scripts/pretrain_foundation_v3.py --epochs 5 --batch-size 256 --device cuda`.
  - epoch 1 loss `0.0603`, epoch 5 loss `0.0547`.
  - artifact: `artifacts_exact_public/foundation_v3_pretrain/fmnet_v3_pretrain.pt`.

v3 Multi-task Fine-tuning
-------------------------
- File: `scripts/train_foundation_v3_multitask.py`.
- Loads pretrained encoder weights when `--pretrained-checkpoint` is supplied.
- Targets (from Stage 2 epsilon=10 dataset):
  - per-decision throughput (log1p MSE on `y_true_mbps` broadcast to all valid decisions).
  - per-decision stop logit (BCE on `stop_label`).
  - final throughput (log1p MSE on `y_true_mbps`, pooled at bucket 99).
  - speed-tier classification (CE on `speed_tier`).
- Default loss weights: prefix throughput `2.0`, stop BCE `1.0`, final throughput `0.5`, speed-tier `0.2`.
- Threshold sweep: 19 thresholds from `0.05` to `0.95`, selection via `policy_constrained_savings` with `min_within_epsilon_rate = 0.66`.
- Hyperparameters:
  - 5 epochs.
  - physical batch size 128, gradient accumulation 8, effective batch size 1024.
  - AdamW, lr `1e-4`, weight decay `0.02`, warmup 500 steps.
  - dropout `0.15`, head dropout `0.10`, gradient clip `1.0`.
- v3 supports an optional second-phase fine-tune via `--phase-1-epochs` and `--phase-2-*-weight` flags so the loss schedule can switch mid-training.

Pivot Plan: Hybrid Foundation-Assisted Stage 2
----------------------------------------------
- Architecture target: a stop-policy model that consumes both
  - foundation trace embedding (from FMNet v3 pretrained encoder, frozen or low-lr fine-tuned), and
  - XGBoost prefix predictions plus their relative-error / uncertainty features at every decision.
- Why this should work:
  - the XGBoost predictions are already strong at every decision (val prefix MAE `18.98`); v3 cannot match them.
  - the foundation trace embedding can supply complementary information (raw trace dynamics, not visible to XGBoost which is window-only).
  - the policy head learns directly on top of accurate prefix predictions, removing the throughput-collapse failure mode that broke v3.
- Inputs per decision `d`:
  - foundation decision-token hidden state at end-bucket `5d+4` (256-dim).
  - XGBoost prefix prediction `y_pred_mbps[d]`.
  - XGBoost relative-error `relative_error[d]` (for training the policy to recognise correctness; not used at deployment time when no ground truth is available).
  - decision metadata: `elapsed_ms`, `observed_buckets_seen`.
- Head: small MLP `[256 + few extra features] -> 128 -> 1` for stop logit. Optional uncertainty head later.
- Training:
  - freeze the FMNet v3 encoder for the first 1-2 epochs (only train the head).
  - then unfreeze with low lr `2e-5` for fine-tuning.
  - target: `stop_label` from the frozen Stage 2 epsilon=10 dataset.
  - selection: `policy_constrained_savings` with `min_within_epsilon_rate = 0.66`.
- Comparison story:
  - Stage 1: keep using foundation v1 (already beats XGBoost on val/test final throughput).
  - Stage 2: hybrid model should match or beat the frozen Stage 2 transformer.
  - the "foundation contribution" is twofold: pretraining-derived trace embeddings used in the hybrid policy, and the standalone v1 throughput regressor.
- Honest framing for paper:
  - foundation pretraining provides measurable benefits in network measurement tasks.
  - a pure-foundation single model cannot beat the specialized two-stage TurboTest pipeline on early termination at this scale.
  - a foundation-assisted hybrid recovers competitive Stage 2 performance while keeping the foundation throughput regressor as a Stage 1 win.
  - this matches the original project proposal language (evaluate whether foundation pretraining provides measurable benefits) without overclaiming pure replacement.

Fallback Direction
------------------
- If A, B, and C all fail, revert to the hybrid foundation-assisted plan documented earlier (`Project-Level Conclusion After Pure Foundation Experiments` block above).
- Hybrid uses XGBoost prefix predictions and uncertainty as explicit inputs to a foundation-assisted policy head.
- The current v1 throughput regressor remains a valid foundation contribution either way.

Clarified Main Project Objective (Single Foundation Two-Stage Model)
====================================================================
- Clarified objective:
  - the project is not simply "raw trace directly to stop decision".
  - the intended model should preserve the TurboTest two-stage order:
    1. Stage 1 predicts throughput at each decision point.
    2. Stage 2 consumes the Stage 1 outputs and decides stop/continue.
  - the research question is whether both stages can be implemented inside one foundation-model system instead of using two separate task-specific TurboTest models.

TurboTest structure:
- Stage 1:
  - model: XGBoost regressor.
  - input: latest 2-second / 20-bucket window features.
  - output: prefix throughput prediction `y_pred_mbps[d]` for each decision `d`.
- Stage 2:
  - model: Transformer stop-policy model.
  - input: sequence of Stage 1 XGBoost predictions and decision metadata.
  - output: stop/continue probability per decision.

Desired foundation-model structure:
- One foundation system should replace the two TurboTest models while keeping the same causal ordering:
  - raw trace / prefix input
  - foundation Stage 1 head predicts prefix throughput at every decision
  - foundation Stage 1 outputs are assembled into a decision sequence
  - foundation Stage 2 policy head consumes that sequence
  - output is stop/continue probability per decision
- In short:
  - `raw trace -> foundation Stage 1 predictions -> foundation Stage 2 stop policy`
- This is different from a one-shot direct classifier:
  - `raw trace -> stop probability`
- The important project claim should be about replacing `XGBoost + Transformer` with a single foundation two-stage system, not about removing the Stage 1 -> Stage 2 logic.

What Previous Experiments Actually Tested
-----------------------------------------
- v1:
  - tested foundation pretraining plus separate downstream heads.
  - Stage 1 throughput regressor worked well on val/test.
  - early-stop head did not explicitly consume the foundation Stage 1 predictions.
  - therefore v1 did not test the exact single-foundation two-stage design.
- v1.5/v1.6/v2/v3:

  - this is close to the objective, but still not exact:
    - the Stage 2 stop head was mostly reading hidden states directly.
    - it was not explicitly forced to consume the model's own Stage 1 output sequence as its main input.
  - these runs show that direct hidden-state stop heads are weak, but they do not fully disprove the intended Stage 1-output-to-Stage 2 foundation design.
- v2.3:
  - attempted XGBoost-prefix distillation.
  - useful diagnostic, but still not the final desired architecture.
- Hybrid plan:
  - uses XGBoost prefix predictions plus foundation embeddings.
  - useful fallback, but not the pure single-foundation replacement because Stage 1 is still TurboTest/XGBoost.

Correct Next Architecture: Foundation Two-Stage FM
--------------------------------------------------
- File target:
  - implement as a new clean script/model path, not by further overloading failed v2/v3 trainers.
  - suggested files:
    - `scripts/foundation_model_twostage.py`
    - `scripts/train_foundation_twostage.py`
- Shared foundation encoder:
  - input: normalized full trace `x_full` with shape `[B, 100, 13]`.
  - use causal bucket-level encoder from FMNet v3:
    - bucket projection `13 -> 256`
    - learned positional embeddings
    - causal Transformer over 100 bucket tokens
    - decision hidden states gathered at buckets `{4, 9, ..., 99}`
  - this preserves the no-future-information constraint.

Stage 1 Foundation Head
-----------------------
- For each decision `d`, using hidden state at end bucket `5d+4`:
  - predict `stage1_mu[d] = log1p(prefix throughput prediction)`
  - optionally predict `stage1_logvar[d]` as uncertainty
- Supervision:
  - primary target: true final throughput `y_true_mbps`, broadcast to every valid decision.
  - loss: log1p MSE or Gaussian NLL.
- Metrics:
  - final throughput MAE/RMSE at decision 19.
  - prefix throughput MAE/RMSE across all valid decisions.
  - compare directly against TurboTest Stage 1 XGBoost predictions stored as `y_pred_mbps[d]`.

Stage 2 Foundation Policy Input
-------------------------------
- Build an explicit decision-level sequence from foundation Stage 1 outputs:
  - `stage1_pred_mbps[d] = expm1(stage1_mu[d])`
  - `stage1_uncertainty[d]` from log variance or absolute confidence proxy
  - `elapsed_ms[d]`
  - `observed_buckets_seen[d]`
  - optionally the decision hidden state `h_decision[d]`
- This sequence is the foundation analogue of the TurboTest Stage 2 input.
- Critical design rule:
  - Stage 2 must consume the Stage 1 foundation outputs explicitly.
  - it should not be only a direct hidden-state classifier.

Stage 2 Foundation Policy Head
------------------------------
- Input sequence per decision:
  - minimal version: `[stage1_pred_log, stage1_uncertainty, elapsed_norm, observed_norm]`
  - stronger version: concatenate the minimal features with `h_decision[d]`
- Policy module:
  - small causal decision Transformer or GRU over 20 decision tokens.
  - output: `stop_logit[d]`.
- Supervision:
  - target: `stop_label[d]` from the valid frozen epsilon=10 Stage 2 dataset.
  - loss: weighted BCE over valid decisions.
- Evaluation:
  - threshold sweep on validation.
  - select threshold by `policy_constrained_savings` with `min_within_epsilon_rate = 0.66`.
  - report F1, within-epsilon rate, emitted-stop rate, mean savings, median savings.
  - compare against frozen Stage 2 epsilon=10 Transformer.

Training Schedule for the Correct Two-Stage Model
-------------------------------------------------
- Phase 0:
  - initialize encoder from FMNet v3 pretraining if available.
  - alternatively run from random init as an ablation.
- Phase 1: Stage 1 foundation training.
  - train encoder + Stage 1 throughput head.
  - optimize prefix throughput prediction only.
  - objective: make foundation Stage 1 outputs as accurate as possible before policy training.
- Phase 2: detached Stage 2 policy training.
  - feed `detach(stage1_mu, stage1_logvar)` into the policy module.
  - train Stage 2 policy head on stop labels.
  - reason: prevent stop-policy loss from destroying Stage 1 regression.
- Phase 3: optional end-to-end fine-tuning.
  - unfreeze the detach boundary.
  - use small learning rate.
  - keep high Stage 1 loss weight.
  - run only if Phase 2 is already competitive.

First Implementation Variant
----------------------------
- Start with the safest version:
  - FMNet v3 causal encoder.
  - Stage 1 head predicts `stage1_mu` and `stage1_logvar`.
  - Stage 2 policy receives detached Stage 1 outputs plus decision metadata.
  - include decision hidden state as an optional flag, default on.
- Loss:
  - `total = lambda_stage1 * prefix_mse + lambda_final * final_mse + lambda_stop * stop_bce`
  - initial weights:
    - Phase 1: `stage1=3.0`, `final=1.0`, `stop=0.0`
    - Phase 2: `stage1=1.0`, `final=0.5`, `stop=1.0`
- Why detach first:
  - previous v3 runs showed unstable throughput when policy and throughput were jointly optimized.
  - detach makes the experiment clean:
    - can the foundation Stage 2 policy use foundation Stage 1 outputs?
    - without corrupting the Stage 1 head.

Success Criteria

- Stage 1:

    - same-or-better mean savings

User-Facing Definition of within-ε
----------------------------------
- For each test:
  - the system stops at some decision time t*
  - the system reports a predicted Mbps value to the user (this is the model's prefix-throughput prediction at t*)
  - y_true_mbps is the ground-truth full-trace throughput
  - the test is "within-ε" if |reported - y_true| / y_true ≤ ε/100
- within-ε rate = fraction of tests in the subset that satisfy this.
- This is the user-facing accuracy of the deployed early-stop system.
- It depends on BOTH the stop policy (when do we stop?) AND the prefix-prediction quality at the stop (what value do we report?).

Per-Decision Prefix Throughput
------------------------------
- "Prefix" = the bucket history available BEFORE a given decision.
- "Decision" = the moments where the early-stop system can choose to stop. In our setup: every 500 ms (buckets 4, 9, 14, ..., 99 -> 20 decision points).
- "Prefix throughput at decision d" = the model's estimate of FINAL throughput, computed using only buckets 0..5d+4.
- "Per-decision prefix throughput MAE" = MAE averaged across all valid (test, decision) pairs.
- Why this matters: when the early-stop system stops at some decision, the user sees the model's prefix-throughput prediction at that decision (NOT at full trace). So per-decision prefix MAE drives the user-facing within-ε at deploy time, not full-trace MAE.

- Method: train Stage 1 with an additional distillation loss that pulls foundation's per-decision predictions toward XGBoost's stored prefix predictions.
